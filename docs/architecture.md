# StylePro Architecture

This document describes the internal architecture, design decisions, and component interactions in StylePro.

## High-Level Overview

StylePro consists of four layers:

```
Browser (JS)          Python (Server-side)
+-----------------+   +-------------------+
| Editor JS       |   | Integration Layer |
| (editor.js,     |<->| (streamlit.py,    |
|  color_picker)  |   |  dash.py,         |
|                 |   |  base.py)         |
+-----------------+   +-------------------+
        |                      |
        v                      v
+-----------------+   +-------------------+
| HTTP API        |<->| Core Layer        |
| (EditorServer)  |   | (Theme, Store,    |
|                 |   |  Permissions)     |
+-----------------+   +-------------------+
                              |
                              v
                      +-------------------+
                      | Storage Adapters  |
                      | (JSON/YAML/SQLite)|
                      +-------------------+
```

## Design Decisions

### CSS Custom Properties for Live Preview

Themes render as CSS custom properties injected once into the document. The JS editor calls `document.documentElement.style.setProperty()` for instant preview with zero Python round-trips during editing. Only on "Save" does the browser POST to the API thread.

```css
/* Generated by theme.to_css() */
:root {
  --sp-bg-color: #ffffff;
  --sp-text-color: #333333;
}
[data-sp-id="btn-a3f2"] {
  --sp-bg-color: #ff5733;   /* per-component override */
}
```

### Per-Component Targeting via data-sp-id

The JS `ElementIdentifier` assigns a deterministic `data-sp-id` attribute to
every element that the user hovers over (lazy, on first hover) and to every
element in a bulk seed pass that runs after the initial page render.

#### ID scheme — global document order, `tag-N` format

An element's ID is constructed as:

```
<lowercase-tag-name>-<zero-based-count>
```

where the count is the element's position among all elements of the same tag
name in document order, counting from zero.

Examples: `button-0` (first `<button>`), `div-3` (fourth `<div>`), `p-12`.

Two groups are excluded from the count:

- **StylePro's own overlay** — the element with `id="sp-overlay-host"` and all
  its descendants.  Excluding the overlay prevents its internal elements from
  shifting the count of app elements.
- **Streamlit's header toolbar** — the element matching
  `[data-testid="stHeader"]` and all its descendants.  Streamlit renders
  navigation controls here that are not part of the app's content.

#### Why this scheme

- **Simple** — a single integer per tag, not a hash of the DOM path.
- **Stable** — for the same app code, Streamlit (and Dash) always render the
  same components in the same document order.  The same element therefore
  receives the same ID on every page load, making saved CSS selectors
  reliable across browser sessions and server restarts.
- **Zero app-code changes** — developers do not need to add IDs or classes to
  their widgets.

#### Stability guarantee and its limits

IDs are stable **as long as the app code does not change the set or order of
rendered widgets**.  Adding, removing, or reordering widgets changes the
document order and can shift counts for every subsequent element of the same
tag.  In that case previously saved `[data-sp-id="..."]` selectors may target
the wrong element.

If you restructure your app or want to start fresh, delete the stale theme
files:

```bash
rm .stylepro/themes/*.json .stylepro/active.txt
```

#### Bulk seeding on page load

`_seedAllIds()` runs once after the framework finishes its initial render (via
a 150 ms debounced `MutationObserver` in Streamlit; immediately after layout
in Dash).  It walks `document.body.querySelectorAll("*")` in document order,
skipping the overlay and header, and assigns `tag-N` IDs in a single O(n)
pass.  This ensures that saved CSS selectors (which reference `data-sp-id`
values) are already present in the DOM when the injected `<style>` block is
parsed by the browser, so theme styles take effect immediately in the normal
(non-edit) view — not only while hovering in edit mode.

### Role-Based Save Separation

- **Admin** saves globally via HTTP POST to the `EditorServer`, which persists to the `ThemeStore` (JSON/YAML/SQLite). These changes affect all users.
- **User** saves locally to the browser's `localStorage` via the `LocalThemeStore` JS module. Personal themes load automatically on return visits but do not affect other users or the server-side store.
- **Guest** cannot edit or save.

This separation prevents non-admin users from modifying global themes while still allowing personal customization.

### Streamlit DOM Injection Strategy

Streamlit presents two challenges for JS injection:

1. **`st.markdown("<script>")` does not execute** — Streamlit strips script tags from markdown for security.
2. **Streamlit reruns the entire Python script** on every user interaction, wiping in-memory JS state.

**Solution:**

- CSS is injected via `st.markdown("<style>...</style>", unsafe_allow_html=True)` into the main document. This works because Streamlit allows style tags.
- JS is injected via `st.components.v1.html(height=0)` — a zero-height iframe. The iframe's script uses `window.parent` to inject `<script>` elements into the parent document (same-origin only, i.e., localhost dev). On cross-origin (Streamlit Cloud), the editor degrades gracefully with a console warning.

### Rerun Resilience

Streamlit reruns wipe in-memory JS state. Mitigation:

- `editor.js` stores `{ edit_mode, unsaved_changes }` in `sessionStorage` keyed by a `session_id` from `st.session_state` (generated once via `uuid.uuid4()`, stable per browser session).
- On each rerun the JS reads `sessionStorage` and restores state.
- The `EditorServer` is started once per session, guarded by `st.session_state["_sp_server_started"]`.

### Server Idempotency

`EditorServer.start()` guards against multiple starts:

- Uses a `threading.Lock` + `is_alive()` check to prevent duplicate threads.
- In Streamlit, additionally gated on `st.session_state["_sp_server_started"]`.
- If the requested port is in use, auto-retries on incrementing ports (up to 10 attempts).
- Port 0 lets the OS assign a free port.

## Component Details

### Core Layer

#### Theme (`stylepro/core/theme.py`)

Two dataclasses:

- **`ThemeVariable`**: A single CSS custom property with `name`, `value`, `label`, `category`, and optional `element_selector` for per-component scoping.
- **`Theme`**: A named collection of `ThemeVariable` entries with metadata. Provides:
  - `to_css()` — renders CSS via `utils.css.theme_to_css()`
  - `to_dict()` / `from_dict()` — JSON-compatible serialization
  - `merge(other)` — immutable union of two themes (other wins on conflict)
  - `apply_patch(patch)` — immutable update of selected variable values

All operations are immutable (return new Theme instances).

#### Store (`stylepro/core/store.py`)

Abstract `ThemeStore` with three adapters:

| Adapter | Storage | Thread Safety | Extra Dependencies |
|---------|---------|--------------|-------------------|
| `JSONThemeStore` | Individual JSON files | `threading.RLock` | None |
| `YAMLThemeStore` | Individual YAML files | `threading.RLock` | PyYAML |
| `SQLiteThemeStore` | Single SQLite database | `threading.RLock` + WAL mode | None (stdlib) |

All adapters implement: `save()`, `load()`, `list_themes()`, `delete()`, `get_active()`, `set_active()`.

#### Permissions (`stylepro/core/permissions.py`)

Table-driven role-permission mapping:

```python
ROLE_PERMISSIONS: dict[Role, set[Permission]] = {
    Role.ADMIN: {EDIT_GLOBAL, EDIT_PERSONAL, SAVE_THEME, DELETE_THEME, VIEW_EDITOR},
    Role.USER:  {EDIT_PERSONAL, VIEW_EDITOR},
    Role.GUEST: set(),
}
```

The table is intentionally module-level and mutable so callers can extend it without patching library internals.

- `check_permission(ctx, perm)` — returns bool
- `require_permission(ctx, perm)` — raises `PermissionError` with descriptive message

### HTTP API Server (`stylepro/editor/server.py`)

Wraps a stdlib `HTTPServer` in a daemon thread. Routes:

| Route | Method | Permission | Description |
|-------|--------|-----------|-------------|
| `/health` | GET | None | Returns `{"status": "ok", "version": "..."}` |
| `/themes` | GET | None | Returns list of theme names |
| `/themes/{name}` | GET | None | Returns theme dict or 404 |
| `/themes` | POST | SAVE_THEME | Save theme (CSS values sanitized) |
| `/themes/{name}/activate` | PUT | EDIT_GLOBAL | Set active theme |
| `/themes/{name}` | DELETE | DELETE_THEME | Delete theme |

All POST body CSS values are sanitized via `sanitize_css_value()` before touching the store. CORS headers are configurable via `allowed_origins`.

### JavaScript Editor (`stylepro/editor/static/editor.js`)

All vanilla JS, single file, runs in the main document. Internal modules:

| Module | Responsibility |
|--------|---------------|
| `ConfigReader` | Reads `window.STYLEPRO_CONFIG`; provides `canEditLayout()` (admin only), `canEdit()` (admin or user) |
| `UndoRedoManager` | 200-action stack with `record()`, `undo()`, `redo()` |
| `CSSVariableManager` | Applies CSS vars live; records to undo stack; persists to sessionStorage; `restore()` on boot |
| `LocalThemeStore` | localStorage persistence for user-role personal themes; `save()`, `load()`, `applyIfPresent()` |
| `ElementIdentifier` | Assigns deterministic `data-sp-id` based on tag + depth + sibling index |
| `EditOverlay` | Shadow DOM host: 8 resize handles + move button + color toolbar; handles/move hidden for non-admin via CSS |
| `MenuIntegration` | MutationObserver watches for Streamlit's hamburger menu popover; injects "StylePro Editor" menu item |
| `SaveManager` | Branches on role: admin calls `_saveGlobal()` (POST to API + activate), user calls `_saveLocal()` (localStorage) |
| `KeyboardHandler` | Escape (deactivate), Ctrl+Z (undo), Ctrl+Y (redo), Ctrl+S (save), arrows (nudge, admin only) |
| `Toast` | Non-intrusive notification messages (success/error) |

#### JS Configuration

`window.STYLEPRO_CONFIG` is injected by Python before `editor.js` loads:

```json
{
  "role": "admin",
  "user_id": "alice@example.com",
  "api_url": "http://127.0.0.1:5001",
  "theme_name": "my_theme",
  "session_id": "uuid-stable-per-session",
  "secret_key": null,
  "fab_position": "bottom-right",
  "css_var_prefix": "--sp"
}
```

### Color Picker (`stylepro/editor/static/color_picker.js`)

Standalone HSV color picker with:
- Canvas-rendered color area and hue slider
- Hex input field
- Recent colors palette (persisted to sessionStorage)
- Singleton pattern exposed as `window.StyleProColorPicker`

### CSS Utilities (`stylepro/utils/css.py`)

All functions are stateless and pure:

- `sanitize_css_value(value)` — security gate; blocks dangerous CSS constructs, raises `ValueError` on match
- `sanitize_variable_name(name)` — enforces `--sp-` prefix, strips invalid characters
- `variables_to_css_block(selector, vars)` — renders a single CSS rule block
- `theme_to_css(theme)` — renders full CSS: `:root {}` for globals, `[selector] {}` for scoped
- `merge_css_blocks(blocks)` — parses and deduplicates CSS blocks (later wins on conflict)

### Integration Layer

#### Base (`stylepro/integrations/base.py`)

Abstract `FrameworkIntegration` class defining the contract:

- `inject()` — called once per render cycle
- `get_css_injection_html(theme)` — returns `<style>` block
- `get_editor_html()` — returns `<script>` block with config + editor JS
- `_build_js_config()` — assembles `window.STYLEPRO_CONFIG` dict
- `_js_config_script()` — returns `<script>` tag setting the config

#### Streamlit (`stylepro/integrations/streamlit.py`)

`StreamlitStylePro` implements the injection strategy described above. Key methods:

- `from_config()` — factory with sensible defaults (JSONThemeStore, guest role)
- `inject()` — starts server (once), generates session_id (once), injects CSS via `st.markdown`, injects JS via `st.components.v1.html(height=0)`
- `get_editor_html()` — builds iframe HTML that injects config + JS + CSS into parent document

#### Dash (`stylepro/integrations/dash.py`)

`DashStylePro` implements the injection strategy for Dash apps. Key methods:

- `from_config()` — factory with sensible defaults (JSONThemeStore, guest role, OS-assigned port)
- `inject(app)` — starts the EditorServer (once, idempotent), registers Flask routes,
  patches `app.index_string`.  Call once after creating the Dash app, before `app.run()`.
- `_register_flask_routes(flask_server)` — adds two routes to the underlying Flask server:
  - `GET /stylepro/editor.js` — serves `color_picker.js` + `editor.js` with
    `window.STYLEPRO_CONFIG` embedded at the top. `Cache-Control: no-cache`.
  - `GET /stylepro/theme.css` — serves the active theme's CSS custom properties.
    Re-reads the store on each request so a newly saved theme is reflected on
    the next full page load. `Cache-Control: no-cache`.
- `_patch_index_string(app)` — modifies `app.index_string` to inject:
  - `<link rel="stylesheet" href="/stylepro/theme.css">` before `{%css%}`
  - `<script src="/stylepro/editor.js"></script>` after `{%scripts%}`

Unlike the Streamlit integration, Dash does not rerun Python on every user
interaction, so there is no per-request injection.  The EditorServer and Flask
routes handle all dynamic content.  The editor JS communicates with the
EditorServer directly for save operations, exactly as in the Streamlit
integration.

## Security

### CSS Value Sanitization

All CSS values submitted through the API pass through `sanitize_css_value()` before storage. Blocked patterns:

- `url()`, `expression()` — prevent resource loading and IE expression evaluation
- `javascript:`, `vbscript:`, `data:` — prevent script protocol injection
- `@import` — prevent external stylesheet loading
- `<script>` — prevent HTML injection
- `behavior:`, `-moz-binding` — prevent browser-specific code execution

Values that pass the blocklist but don't match the safe-value allowlist generate a warning log but are still accepted (to avoid false positives on valid but uncommon CSS values).

### CORS

The `EditorServer` supports configurable `allowed_origins` (default: `["*"]`). In production, restrict this to your app's origin.

## Known Limitations

- **Streamlit Cloud**: Cross-origin iframe restrictions prevent JS injection into the parent document. The editor will not function on Streamlit Cloud deployments (documented limitation, workaround planned for v1.1).
- **data-sp-id instability**: If Streamlit reruns reorder the DOM, previously assigned `data-sp-id` values may not match. Styles will load but may not apply to the intended elements.
- **Single active theme**: Only one theme can be active at a time across all users. User-role localStorage themes are independent but don't participate in the server-side active theme system.
